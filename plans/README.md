## Findings considered and rejected (do not re-audit)

- **"Cache warming has no already-warm guard" (cold-start auditor)** — false; guards exist at `server/index.ts:1132/1160/1179`.
- **Per-request dynamic `import("./redis.js")` overhead** — negligible; Node caches modules after first load.
- **CurrencyContext value not memoized** — provider is the app root and only re-renders when currency/rates change, which is exactly when consumers must re-render; memoization is cosmetic.
- **HTTP Cache-Control on /api responses** — rejected: `/api/trade-ups` is tier-dependent (free-tier delay); browser/CDN caching risks cross-tier leakage for marginal gain (see plan 006 scope).
- **58 CREATE INDEX IF NOT EXISTS statements at boot** — catalog no-ops (~ms each); the real boot cost was the two full-table COUNT scans, fixed by plan 008's version gate.
- **localStorage JSON.parse size limits** — micro-optimization.
- **User tier cache TTL 60s "too long"** — by design; explicit invalidation hooks already exist (`invalidateUserCache`).
- **Move `@fontsource/inter` "verify which is used"** — verified: Geist only; inter removed in plan 002.
- **better-sqlite3 session store blocking** — WAL is enabled (`server/auth.ts:20`), per-request ops are single-row point lookups; fine at current scale.
- **react-day-picker → native date inputs** — softened to lazy-loading (plan 003) to avoid a UX change nobody asked for.
- **List virtualization (TradeUpTable/SkinList)** — pagination already caps row counts; revisit only if pages grow past ~200 rows.
- **CSP nonce instead of 'unsafe-inline'** — real hardening but M effort with Vite plumbing; not a perf item, no current XSS sink found. Revisit in a security pass.
- **Client-side per-tab fetch cache / react-query** — over-engineering for this codebase; server-side single-flight (006) addresses the cost.
